Absorbathon is a 2D, side scrolling, arcade-esque game made using Unity.

Role:

Programmer.

State:

Released.

Project length:

8 Weeks.

About:

While working on Absorbathon I mainly focused on enemy ai and procedural level generation.
The enemy ai uses a finite state machine and follows through various set states depending on the type of the ai (attack, flee). They use vision cones to be able to detect the player and then can either enter attack or flee states depending on the ai type.
For procedural level generation I used a grid system that two random points on the grid are generated that are between a specified minimum and maximum Manhattan distance, then a path finding algorithm is used to generate a path between the start and end. Then the next step is to loop through all the path cells and do a check to see if a random walk should take place each step of the walk decreases the chance of the next step happening. Once the random walk is finished the path finding algorithm is used to generate a path from the end of the walk to the midpoint between the start of the walk and the end of the main path, this is the repeated for the remaining main path cells. The use of these systems together generates branching dungeons with few dead ends and a medium density of rooms.

Looping Halls is a tech demo project that experiments and showcases world looping and non euclidean geometry in Unity.

Role:

Solo developer.

State:

Released.

Project length:

1 Week.

About:

Looping Halls is a tech demo that has the player placed in a small corridor like environment where they are free to walk around. There is a central room with 2 branching corridors going down either corridor will cause the player to loop back to the beginning hallway. When going down the right hallway there is a 25% chance it will infinitely loop util the player turns around and goes back the way they came where they will then go back into the central room. As the player keeps moving through loops of the environment items will appear and disappear, for example on the 2nd loop a chair will appear in the central room and on the 3rd loop a section of wall on the right side of the central room will disappear revealing a 3rd corridor. The looping trick is achieved through the use of a 2nd camera and render planes, when the player touches certain triggers, it will cause the 2nd camera to start mimicking the rotation and position but in a different part of the environment. The camera's view is then projected onto a render plane so that it appears as one seamless corridor even though its two completely separate parts of the map. When the player touches the render plane they are then teleported to where the second camera was so they appear to have not move at all.

FGE is a 2D game engine that I developed using C++.

Role:

Solo developer.

State:

Complete.

Project length:

22 Weeks.

About:

FGE is the first game engine and editor that I made. I made it during my second year of university.
It features a working editor that consists of:
  • A save/load system
  • Profiling tools; that allows the user to see how much time profiled function calls are taking and to see a flame graph representation for each update loop.
  • An inspector; that allow the inspecting of game objects to see their transform and components.
  • An asset window; that allows importing and use of bitmap assets
  • A scene hierarchy window; to allow the user to see what game objects are in a scene and how they relate to one another.
The engine features:
  • A game object system similar to unity's; game objects have a transform and a list of components attached to it as well as a parent and list of child game objects.
  • A scene system where game objects are place within the scene the during the update loop all the objects in the active scene are updated.
  • A collision/physics system that uses axis aligned bounding boxes to detect collisions and fire event when to colliders are overlapping.
  • Image rendering so the user can add bitmaps to be used in the editor which can then be displayed.
  • A logging system so that any errors, warnings or information logs created by the engine or user can be logged to console and saved to a file.
  • Multithreading so there is little to no preformance hit when writing the log messages to files.

External libraries:

SDL, Dear ImGui and pugixml.
